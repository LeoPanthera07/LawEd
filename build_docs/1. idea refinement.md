# NyayaSetu — Polished Idea Document

## 1. Problem Statement

**The business problem:** India has 1.4 billion people and one of the most complex legal systems in the world. In 2023–24, three foundational laws were replaced — IPC became BNS, CrPC became BNSS, and the Indian Evidence Act became BSA. Most citizens have no awareness of these laws, no access to affordable legal guidance, and no structured way to initiate legal action. Filing an FIR at a police station, approaching a lawyer, or understanding one's rights requires either money, connections, or both. The result: the majority of India's legally actionable incidents go unreported, unacted upon, or mishandled — not because citizens don't want justice, but because the path to it is inaccessible.

**Target users:** Primary — any Indian adult who has experienced or anticipates a legal situation, regardless of education, income, or legal literacy. Secondary — evaluator lawyers on the platform who validate escalated cases. Tertiary — law firms and police stations as institutional partners in later phases.

**Why this matters now:** The transition from IPC/CrPC/Evidence Act to BNS/BNSS/BSA is creating a knowledge vacuum — even practicing lawyers are still navigating the new framework. This is the ideal window to establish NyayaSetu as the definitive intelligence layer over the new legal corpus before incumbents catch up. The timing is structural, not opportunistic.

## 2. Solution Overview

**What the product does:** NyayaSetu is a web-first legal intelligence platform. A citizen describes their situation once in plain natural language — in English, Hindi, or Hinglish. The system, powered by a LangGraph agentic pipeline grounded in BNS, BNSS, and BSA, returns a structured legal intelligence response: the most relevant clauses and articles, exact statutory text, plain-English explanation, and contextual relevance per clause. For situations requiring it, the system generates a paid structured writeup — FIR-ready or lawyer-ready — that the citizen can walk into any police station or law firm with directly. High-stakes or low-confidence cases are escalated to a verified evaluator lawyer via a role-based dashboard.

**Core value proposition:** NyayaSetu is the digital entry point to the Indian justice system — the infrastructure layer that sits between every Indian citizen and every legal action they will ever need to take.

**Key features — prioritized:**

MVP (Phase 1):

- Natural language incident input (English, Hindi, Hinglish)
- Agentic clause mapping via LangGraph + RAG over BNS, BNSS, BSA
- GraphRAG-powered clause relationship traversal via Neo4j
- Structured legal intelligence output (top 5–10 clauses, exact text, explanation)
- Paid structured writeup generation (FIR-ready / lawyer-ready document)
- Human escalation node — high-stakes or low-confidence cases routed to evaluator lawyer dashboard
- RBAC — Citizen, Evaluator Lawyer, Admin roles
- Per-request pricing for clause mapping, per-writeup pricing for structured documents

Phase 2:

- Unique case ID system — citizen shares ID with law firm for instant structured intake
- State amendment overlay — state-specific legal variations as additional retrieval layer
- Multilingual expansion beyond Hindi/Hinglish
- Law firm partner onboarding

Phase 3:

- Agentic plaintiff/defendant debate with judge agent for case strength scoring
- Indian judiciary public records integration
- Fine-tuned LLM for evaluator lawyer role
- Full complaint registration pipeline to police stations and courts

## 3. Technical Architecture

**Stack:**

- Frontend: Next.js (deployed on Vercel)
- Backend: FastAPI (deployed on Railway)
- Orchestration: LangGraph (agentic pipeline, multi-node, human-in-the-loop support)
- Vector DB: Qdrant (self-hosted on Railway for MVP, Qdrant Cloud at scale)
- Graph DB: Neo4j Community Edition (clause relationship graph, GraphRAG traversal)
- LLM: Groq + Llama 3 (MVP, free tier) → Anthropic Claude API (production)
- Embeddings: multilingual-e5-large (open source, self-hosted, handles Hindi/Hinglish)
- Relational DB: PostgreSQL (users, payments, case records, RBAC)
- Payments: Razorpay (INR-native, citizen-friendly)

**Data flow:**

1. Citizen submits natural language input via Next.js frontend
2. FastAPI receives request, initiates LangGraph pipeline
3. Intent and language detection node identifies act scope and translates if needed
4. Query expansion node generates multiple retrieval queries from the input
5. Qdrant retrieval node fetches top-k clause chunks via multilingual-e5-large embeddings
6. Neo4j graph traversal node expands results by traversing clause relationships — related sections, cross-act references, procedural dependencies
7. Confidence evaluator node scores the mapping — low confidence or high-stakes flag triggers human escalation node, routing to evaluator lawyer dashboard
8. Response synthesis node (Groq/Claude) generates structured legal intelligence output grounded strictly in retrieved clauses — no hallucination outside retrieved context
9. Output returned to citizen UI — clause map displayed, writeup generation offered as paid action
10. All case records, user data, and payment events written to PostgreSQL

**AI/ML components:**

- multilingual-e5-large: embedding model for Hindi/Hinglish/English clause and query encoding
- LangGraph: multi-node agentic orchestration with conditional edges (confidence routing, escalation routing, writeup trigger)
- GraphRAG via Neo4j: automated clause relationship extraction run pre-launch using LLM pass over full BNS/BNSS/BSA corpus — relationships stored as graph, traversed at query time
- Groq + Llama 3 / Claude API: response synthesis, writeup generation, confidence scoring
- Pre-launch pipeline: LLM-automated extraction of cross-references, procedural dependencies, and semantic relationships between all clauses across three acts — stored in Neo4j before first user query

## 4. Prototype Plan

**Phase 0 — Data pipeline (Days 1–3):**

- Convert BNS, BNSS, BSA PDFs to structured JSON (section → subsection → clause level)
- Chunk and embed using multilingual-e5-large, store in Qdrant
- Run LLM-automated relationship extraction pass over full corpus, populate Neo4j graph
- Validate retrieval quality on 20 hand-crafted test queries before moving forward

**Phase 1 — Core pipeline (Days 4–7):**

- Build LangGraph pipeline: intent detection → query expansion → Qdrant retrieval → Neo4j traversal → confidence evaluation → response synthesis
- FastAPI endpoints: /query, /writeup, /escalate
- Unit test each LangGraph node independently
- Milestone: end-to-end query returns grounded clause map

**Phase 2 — Frontend + Auth (Days 8–12):**

- Next.js frontend: input interface, structured results display, writeup purchase flow
- PostgreSQL schema: users, cases, payments, roles
- RBAC implementation: Citizen, Evaluator Lawyer, Admin
- Evaluator lawyer dashboard: case queue, review interface, approve/escalate actions
- Razorpay integration for writeup payments
- Milestone: citizen can submit query, see results, purchase writeup, lawyer can review escalated cases

**Phase 3 — Hardening (Days 13–16):**

- Hallucination guard: response synthesis node constrained strictly to retrieved context, citation required for every clause reference
- Confidence scoring calibration: define thresholds for auto-escalation
- Human escalation flow tested end-to-end with a real lawyer volunteer
- Milestone: zero hallucinated clause references across 50 test queries

**Phase 4 — Deploy + validate (Days 17–20):**

- Deploy: Vercel (frontend), Railway (FastAPI, Qdrant, Neo4j, PostgreSQL)
- Soft launch to 10–20 trusted users (friends, IIT Gandhinagar network)
- Collect feedback on output quality, UX, and writeup usefulness
- Lawyer volunteer validates 10 escalated cases for accuracy
- Milestone: first paid writeup generated by a real user

## 5. Business Validation

**Market size:** India has approximately 400 million legally active adults who interact with the justice system in some capacity annually — FIRs, civil disputes, property cases, workplace incidents, consumer complaints. At even ₹10/request and 1% penetration that is ₹400 million/year from query pricing alone. Writeups at ₹99–299 each with 0.1% penetration represent a separate revenue stream in the same order of magnitude. The TAM is not a SaaS market — it is a legal infrastructure market for the world's most populous democracy.

**Competition:** No direct competitor currently combines BNS/BNSS/BSA-grounded RAG + GraphRAG + agentic pipeline + citizen-grade UX + structured writeup generation in one product. Existing players — LawRato, Vakil Search, MyAdvo — are lawyer marketplaces, not legal intelligence tools. AI legal tools in India are either generic LLM wrappers or English-only research tools for lawyers, not citizens.

**Unique value proposition:** NyayaSetu is the only system that understands a citizen's situation in their own language, maps it to the correct new Indian laws with grounded precision, and produces a document they can act on immediately — without needing a lawyer to begin the process.

**Risks and mitigations:**

|Risk|Mitigation|
|---|---|
|LLM hallucination on legal references|Strict retrieval grounding — synthesis node cannot reference any clause not in retrieved context. Citation enforced at node level.|
|Low citizen trust in AI for legal matters|Lawyer escalation path, exact statutory text always shown, no black-box outputs|
|BNS/BNSS/BSA corpus changes via amendments|Versioned corpus pipeline — amendments trigger re-embedding and graph re-extraction automatically|
|Regulatory risk — practicing law without a license|NyayaSetu provides legal information, not legal advice. Writeups are structured incident summaries, not legal opinions. Disclaimer enforced at every output.|
|Low willingness to pay at ₹10/request|Free tier builds habit and user base. Paid writeup is the first revenue event — value is obvious (document they can physically use).|

## 6. Constraints and Assumptions

**Assumptions:**

- BNS, BNSS, and BSA PDFs are parseable into clean structured JSON at section/clause level
- multilingual-e5-large embeddings provide sufficient retrieval quality for Hindi/Hinglish legal queries
- Groq free tier is sufficient for MVP query volumes
- Citizens are willing to describe their situation in text (voice input deferred to Phase 2)
- NyayaSetu's output constitutes legal information, not legal advice — this legal distinction protects the platform from practicing-law-without-a-license liability
- One evaluator lawyer volunteer is sufficient for Phase 1 escalation validation

**Constraints:**

- Open source and free infrastructure only for MVP — no paid cloud AI APIs until revenue justifies it
- English, Hindi, and Hinglish only for MVP — regional languages in Phase 2
- Central laws only for MVP — state amendments deferred to Phase 2
- No voice input for MVP — text only
- Human escalation handled manually via dashboard for MVP — no automated lawyer matching yet
- Solo builder with session-based Claude Code development — architecture must be modular so each session builds one clean component
