## 1. System Overview

NyayaSetu is a multi-component, agentic AI system. At its core it is a retrieval-augmented generation pipeline orchestrated by LangGraph, sitting behind a FastAPI backend, served to a Next.js frontend. It draws from three data stores — Qdrant (vector search), Neo4j (graph traversal), and PostgreSQL (relational records) — and calls an external LLM (Groq/Claude) for synthesis and generation.

The system must handle two fundamentally different workloads simultaneously:

- **Latency-sensitive citizen queries** — natural language in, structured legal intelligence out, in under 15 seconds
- **Async human escalation workflows** — cases routed to a lawyer dashboard, reviewed, and resolved within 4 hours

The architecture is designed to keep these two workloads decoupled, independently scalable, and failure-isolated from each other.

## 2. Architecture Principles

|Principle|Application in NyayaSetu|
|---|---|
|Separation of concerns|Query pipeline, writeup generation, escalation, auth, and payments are independent modules — failure in one does not cascade to others|
|Retrieval-grounded output only|LLM synthesis node receives only retrieved context — it cannot access the internet or generate references outside the retrieved corpus|
|LLM provider agnosticism|LLM is called via an abstraction layer — switching from Groq to Claude API requires a single config change|
|Stateless backend|FastAPI services are stateless — all state lives in PostgreSQL, Qdrant, or Neo4j — enabling horizontal scaling|
|Modular agentic pipeline|Each LangGraph node is independently testable and replaceable — a node can be upgraded, retested, and redeployed without rebuilding the entire pipeline|
|Human-in-the-loop by design|Escalation is a first-class architectural concept — not an afterthought — with its own async workflow, notification system, and dashboard|
|Free-tier first|All infrastructure components selected with free or open-source tiers as the MVP deployment target|

## 6. Data Architecture

### 6.1 PostgreSQL — Relational Data

Handles all structured business data: users, sessions, cases, payments, roles, writeups. This is the system of record. Every event that matters to the business is persisted here.

**Core tables (summary):**

|Table|Purpose|
|---|---|
|`users`|Citizen, Lawyer, Admin accounts — credentials, roles, metadata|
|`sessions`|JWT refresh token tracking, session invalidation|
|`queries`|Every query submitted — input text, language detected, pipeline status|
|`clause_maps`|Output of every completed pipeline run — top clauses, scores, JSON|
|`escalations`|Escalated cases — reason, status, assigned lawyer, resolution|
|`writeups`|Generated writeup records — citizen details, PDF path, payment ID|
|`payments`|Razorpay payment records — order ID, amount, status, type|
|`lawyer_reviews`|Evaluator lawyer actions on escalated cases — approval, edits, notes|
|`audit_log`|All admin actions, role changes, system events|

---

### 6.2 Qdrant — Vector Data

Stores embeddings of every clause chunk from BNS, BNSS, and BSA. Each vector is accompanied by a metadata payload enabling filtered retrieval.

**Collection structure:**

```
Collection: nyayasetu_clauses

Vector dimensions: 1024 (multilingual-e5-large output)
Distance metric: Cosine similarity

Payload schema per vector:
{
  "act": "BNS" | "BNSS" | "BSA",
  "chapter": "string",
  "section_number": "string",
  "section_title": "string",
  "clause_text": "string",
  "chunk_index": integer,
  "language": "en",
  "version": "1.0"
}
```

**Chunking strategy:**

- Each section is a primary chunk
- Sub-sections longer than 512 tokens are split with 50-token overlap
- Section title and act name are prepended to every chunk for context
- Estimated total vectors: 8,000–15,000 across BNS + BNSS + BSA

---

### 6.3 Neo4j — Graph Data

Stores the clause relationship graph extracted by the pre-launch LLM pipeline. Enables GraphRAG traversal — finding related clauses that vector search alone would miss.

**Node types:**

```
(:Clause {
  id: "BNS-103",
  act: "BNS",
  section: "103",
  title: "Punishment for murder",
  text: "...",
  chapter: "VI"
})
```

**Relationship types:**

|Relationship|Meaning|Example|
|---|---|---|
|`REFERENCES`|Clause explicitly references another|BNS 103 → REFERENCES → BNS 100|
|`PROCEDURAL_DEPENDENCY`|One clause requires action under another|BNS 103 → PROCEDURAL_DEPENDENCY → BNSS 173|
|`EVIDENCE_REQUIREMENT`|Clause requires evidence governed by BSA|BNS 103 → EVIDENCE_REQUIREMENT → BSA 57|
|`CROSS_ACT_LINK`|Semantic connection across acts|BNSS 35 → CROSS_ACT_LINK → BNS 150|
|`EXCEPTION_OF`|One clause is an exception to another|BNS 101 → EXCEPTION_OF → BNS 100|

**Pre-launch graph population pipeline:**

```
1. Load all clause chunks from structured JSON
2. For each clause, call LLM with prompt:
   "You are a legal expert analysing Indian law.
    Given this clause: [clause text]
    Identify all relationships to other clauses in BNS, BNSS, and BSA.
    Return as JSON: [{source, target, relationship_type, reasoning}]"
3. Parse LLM output, validate section numbers against corpus
4. Write valid relationships to Neo4j
5. Log all relationships for manual audit
```

### 7.1 API Design Principles

- RESTful, versioned under `/api/v1/`
- All responses follow a consistent envelope schema
- Authentication via Bearer JWT on all protected routes
- Rate limiting enforced at the API gateway layer

**Standard response envelope:**

```json
{
  "success": true,
  "data": { },
  "error": null,
  "meta": {
    "request_id": "uuid",
    "timestamp": "ISO8601",
    "version": "1.0"

## 8. Authentication and RBAC Architecture

### 8.1 Token Strategy

```
Access Token:  JWT, 15-minute expiry, contains user_id + role
Refresh Token: JWT, 7-day expiry, stored in HttpOnly cookie
Token rotation: New refresh token issued on every refresh call
Blacklisting:  Invalidated tokens stored in PostgreSQL sessions table

## 9. Pre-Launch Data Pipeline

Before the first citizen query is processed, three offline pipelines must be run and validated:

### 9.1 PDF Ingestion Pipeline

```
BNS PDF + BNSS PDF + BSA PDF
        │
        ▼
  PDF Parser (PyMuPDF)
        │
        ▼
  Structure Extractor
  (Chapter → Section → Subsection → Clause)
        │
        ▼
  JSON Schema Validator
  (every clause has: act, chapter, section_number,
   section_title, clause_text)
        │
        ▼
  Chunker (512-token max, 50-token overlap,
           title prepended)
        │
        ▼
  Quality Check: 20 hand-crafted test clauses
  verified against source PDF
```

### 9.2 Embedding and Indexing Pipeline

```
Clause Chunks (JSON)
        │
        ▼
  multilingual-e5-large Encoder
  (batch processing, GPU if available)
        │
        ▼
  Qdrant Upsert
  (vector + payload per chunk)
        │
        ▼
  Retrieval Validation:
  20 test queries → verify expected clauses
  appear in top-10 results
```

### 9.3 Graph Extraction Pipeline

```
All Clause Chunks
        │
        ▼
  LLM Relationship Extractor
  (Groq/Claude — one clause at a time)
        │
        ▼
  JSON Parser + Section Number Validator
  (reject any relationship referencing
   a section not in the corpus)
        │
        ▼
  Neo4j Batch Writer
        │
        ▼
  Graph Validation:
  Spot-check 20 known clause relationships
  against lawyer review
```

## 10. Scalability Strategy

### 10.1 MVP Scale Targets

|Metric|MVP Target|Architecture Headroom|
|---|---|---|
|Concurrent users|100|FastAPI async handles 500+|
|Queries per day|1,000|Qdrant handles 100K+/day on 2 CPU|
|Qdrant vector count|~15,000|Qdrant handles 10M+ vectors|
|Neo4j relationships|~50,000|Neo4j Community handles 1B+|
|PostgreSQL rows|~500,000|PostgreSQL handles 100M+|

### 10.2 Scale-Up Path

|Trigger|Action|
|---|---|
|Query latency p95 > 20 seconds|Add FastAPI worker replicas on Railway|
|Qdrant memory pressure|Migrate to Qdrant Cloud (managed, pay-as-you-go)|
|Neo4j memory pressure|Migrate to Neo4j AuraDB (managed free tier → paid)|
|LLM cost > 30% of revenue|Evaluate fine-tuned smaller model for synthesis node|
|PostgreSQL connection saturation|Add PgBouncer connection pooler|
|10,000+ concurrent users|Migrate FastAPI to Kubernetes on GCP/AWS|

## 11. Reliability and Failure Handling

### 11.1 Failure Modes and Responses

|Component|Failure Mode|System Response|
|---|---|---|
|LLM API (Groq/Claude)|Timeout or rate limit|Retry with exponential backoff × 3, then return graceful error to citizen|
|Qdrant|Connection failure|Return cached top clauses if available, else graceful error|
|Neo4j|Connection failure|Skip graph traversal, proceed with Qdrant results only — degrade gracefully|
|PostgreSQL|Connection failure|Return 503, alert Admin via email|
|Razorpay|Payment verification failure|Do not generate writeup, return clear error, log for manual review|
|LangGraph node|Unhandled exception|Catch at pipeline level, log full trace, return graceful error to citizen|

### 11.2 Data Persistence Guarantee

Every query and result is written to PostgreSQL before the response is returned to the citizen. If the frontend fails to display the result, the citizen can retrieve it from their case history. No query result is ever lost.

## 13. Security Architecture Summary

|Layer|Control|
|---|---|
|Transport|TLS 1.3 on all connections — Vercel and Railway enforce this by default|
|Authentication|JWT — access token 15 min, refresh token 7 days, HttpOnly cookie|
|Authorisation|RBAC enforced at FastAPI route level via dependency injection|
|Input validation|Pydantic models on all API inputs — no raw string injection possible|
|Rate limiting|SlowAPI middleware on FastAPI — 10 req/min unauthenticated, 30 authenticated|
|Payment security|Razorpay signature verification on every payment callback|
|Data privacy|DPDP Act 2023 compliant — data minimization, right-to-erasure endpoint|
|Secret management|Environment variables via Railway — never hardcoded|
|Audit logging|All admin actions and role changes logged to `audit_log` table|
